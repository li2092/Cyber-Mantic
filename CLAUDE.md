## 关于产品
- 核心理论：根据用户提供的信息，智能选择3-5个最合适的术数理论进行分析，解决理论间的冲突矛盾，最终结合MBTI特点生成个性化报告
- 呈现形式分为两种：问道（对话预测），推演（输入信息预测）
- 围绕着两各核心功能衍生出：典籍（阅读模块）、用户画像（使用情况）、历史记录（存储预测记录）、设置（主题和API接口设置）
- 涉及术数理论：八字、奇门遁甲、梅花易数、六壬、六爻、紫微、小六壬、测字术
- 特点：最小信息输入，保护隐私

## 关于开发过程
- 与传统开发过程严格按照PRD将预设功能编程编码不同，作者以客户视角提出的主要为功能实现或抽象概念
- 把Markdown文件当作AI的硬盘记忆，强制AI遵循“三文件模式”：
  - task_plan.md （任务规划书）：记录目标、阶段、进度和错误
  - notes.md （笔记/草稿）：存储调研结果、中间思考和长文本
  - [deliverable].md （交付物）：最终产品，交付物文档命名要清晰，内容要定期更新，修改到哪部分代码时要阅读相关文件
  - 工作流循环：读计划→行动→记录→复盘，避免普通AI Agent“接收指令→执行→迷失→崩溃”的问题
- 另外，需要将功能或抽象概念落地需要补充技术落地的介绍文档，不用参考传统开发模板，主要视角是以数据流、系统交互、逻辑等连贯性关键节点进行表达
- 或者用你觉得舒服的表达方式进行技术落地介绍
- 请告诉自己：我是没有记忆的，所有要点都要用文件记下来、保存下来，特别是设计理念和系统相互的联系，不要改一个忘一个
- 好记性不如烂笔头
## 三文件模式操作规范

### 文件命名规范
```
docs/
├── [功能名]_task_plan.md    # 任务规划书
├── [功能名]_notes.md        # 笔记/草稿
└── [功能名]_[交付物名].md   # 交付物文档
```

**示例**：问道界面完善
- `wendao_task_plan.md`
- `wendao_notes.md`
- `wendao_enhancement.md`

### 文件存放位置
- **三文件系统**：统一存放在 `docs/` 目录
- **代码文件**：按现有项目结构存放在 `cyber_mantic/` 下
- **Prompt模板**：存放在 `cyber_mantic/prompts/` 下

### 各文件内容要求

**task_plan.md（任务规划书）**
- 目标：明确本次任务要达成什么
- 当前状态：已完成/进行中/待处理的问题清单
- 阶段规划：P0/P1/P2优先级划分
- 关键文件清单：涉及的代码文件路径
- 错误日志：已解决和待解决的问题
- 下一步行动：具体可执行的任务

**notes.md（笔记/草稿）**
- 架构理解：数据流、模块关系
- 技术细节：关键代码片段、实现方案
- 已修复问题详解：问题原因和解决方案
- 待探索方向：优化点、潜在问题
- 测试用例：正常流程和异常流程

**[deliverable].md（交付物）**
- 功能概述：做什么、为什么
- 交互流程：用户视角的操作流程
- 详细设计：各模块/阶段的输入输出
- UI/API规格：组件尺寸、接口格式
- 异常处理：边界情况的处理策略
- 后续规划：短期/中期/长期路线图

### 工作流循环
```
1. 读计划 → 了解当前状态和目标
2. 行动   → 编码、修复、测试
3. 记录   → 更新三文件（尤其是问题和进度）
4. 复盘   → 检查目标是否达成，规划下一步
```
